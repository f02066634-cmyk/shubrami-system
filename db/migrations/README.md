# مجلد الهجرات (migrations)

هذا المجلد لتسجيل **التغييرات التدريجية** على بنية القاعدة من الآن فصاعداً.

بينما تمثّل ملفات `db/schema.sql` و`functions.sql` و`policies.sql` و`triggers.sql`
**الحالة الكاملة الراهنة** للقاعدة (لقطة)، تمثّل ملفات هذا المجلد **الخطوات
المتتابعة** التي أوصلتنا إليها — كل ملف تغيير واحد قابل للتطبيق مرّة واحدة.

---

## متى تضيف ملف هجرة؟

عند أي تعديل على بنية Supabase: إضافة/تعديل عمود، دالة، سياسة، trigger، قيد،
فهرس… أنشئ ملف هجرة يصف التغيير بـ SQL قابل للتنفيذ.

## التسمية

استخدم بادئة زمنية قابلة للفرز + وصف مختصر بالإنجليزية:

```
migrations/
  20260722_1000_add_column_x_to_shops.sql
  20260722_1130_new_policy_debts_delete.sql
```

## قالب ملف الهجرة

```sql
-- الهجرة: <وصف موجز>
-- التاريخ: YYYY-MM-DD
-- السبب: <لماذا هذا التغيير>

-- === التطبيق (Up) ===
ALTER TABLE public.shops ADD COLUMN example text;

-- === التراجع (Down) — اختياري لكن مُستحسَن ===
-- ALTER TABLE public.shops DROP COLUMN example;
```

---

## سير العمل (workflow) مع كل تغيير

1. طبّق التغيير في Supabase (أو نفّذ ملف الهجرة عليه).
2. أضِف ملف الهجرة هنا.
3. **حدِّث ملفات اللقطة** في `db/` (`schema.sql` / `functions.sql` /
   `policies.sql` / `triggers.sql`) لتعكس الحالة الجديدة.
4. اعمل commit للاثنين معاً (الهجرة + تحديث اللقطة) — تطبيقاً للقاعدة الذهبية:
   **أي تغيير في Supabase = commit مقابل في `db/`**.

> اللقطة تُجيب: «كيف تبدو القاعدة الآن؟» — والهجرات تُجيب: «كيف وصلنا إلى هنا؟».
> نحتفظ بالاثنين معاً.
