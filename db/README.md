# توثيق قاعدة البيانات — مرآة Supabase

هذا المجلد هو **مرآة نصّية ملتزَمة في git لبنية قاعدة بيانات Supabase** الخاصة
بنظام أسواق الشبرمي. الغرض منه:

1. **مرجع الاستعادة** — إن ضاعت القاعدة أو تلِفت، تُعاد بناؤها من هذه الملفات.
2. **سجلّ التغييرات** — كل تعديل على البنية (جدول/دالة/سياسة/trigger) يُوثَّق هنا
   عبر commit، فيصبح لدينا تاريخ واضح لِمَن غيّر ماذا ومتى.
3. **مصدر الحقيقة للمراجعة** — يمكن قراءة البنية كاملةً دون فتح لوحة Supabase.

> ⚠️ **هذه الملفات لا تُنفَّذ تلقائياً على القاعدة.** Supabase هو النظام الحيّ؛
> وهذا المجلد نسخة موثّقة تعكسه. المزامنة **يدوية** ومسؤولية المطوّر (انظر
> القاعدة الذهبية أدناه).

---

## القاعدة الذهبية

> **أي تغيير في Supabase = commit مقابل في `db/`.**

عند تعديل أي شيء في القاعدة (إضافة عمود، تعديل دالة، إنشاء سياسة، إسقاط
trigger…): حدِّث الملف المقابل في هذا المجلد **في نفس الـ commit** (أو أضف ملف
هجرة في `migrations/` — انظر `migrations/README.md`). هكذا تبقى المرآة صادقة.

## المحتويات

| الملف | المحتوى |
|------|---------|
| `schema.sql`    | الجداول (11) + القيود (PK/FK/UNIQUE/CHECK) + الفهارس + تفعيل RLS |
| `functions.sql` | الدوال (15): المساعِدة + حُرّاس المصروفات + حُرّاس السندات + دوال RPC |
| `policies.sql`  | سياسات RLS (33 سياسة، للدور `public`) |
| `triggers.sql`  | مشغّلات المصروفات (2) + مشغّلات السندات (2) + مشغّل `auth.users` (`on_auth_user_created`) |
| `migrations/`   | ملفات الهجرة التدريجية من الآن فصاعداً (انظر README بداخله) |

## ترتيب التنفيذ للاستعادة الكاملة

الترتيب **إلزامي** لأن كل طبقة تعتمد على ما قبلها:

```
1) schema.sql      ← الجداول والقيود والفهارس وتفعيل RLS
2) functions.sql   ← الدوال (is_admin, has_tab, الحُرّاس, دوال RPC)
3) policies.sql    ← السياسات (تستدعي is_admin() و has_tab())
4) triggers.sql    ← المشغّلات (تستدعي دوال الحُرّاس)
```

**لماذا هذا الترتيب بالضبط:**

- **الجداول أولاً**: القيود المرجعية (FK) تحتاج الجدول المرجوع إليه موجوداً.
  الجداول داخل `schema.sql` نفسها مرتّبة بالتبعية:
  `profiles → bank_accounts → expense_categories → expenses →
  bank_account_assignments → expense_category_assignments → shops →
  transactions → debts → installments → audit_log`.
- **الدوال قبل السياسات**: كل سياسات RLS تقريباً تستدعي `is_admin()` أو
  `has_tab(text)` — يجب أن تكون الدالتان موجودتين أولاً.
- **الدوال قبل المشغّلات**: المشغّلان `trg_guard_expense_*` ينفّذان
  `guard_expense_insert()` و`guard_expense_immutable_columns()`.
- **دوال RPC تعتمد على الجداول** (مثل `rpc_record_receipt` التي تستدعي
  `rpc_next_receipt` وتكتب في `transactions`/`shops`/`debts`/`installments`)،
  لذا تأتي بعد `schema.sql`.

مثال تنفيذ عبر psql:

```bash
psql "$DATABASE_URL" -f db/schema.sql
psql "$DATABASE_URL" -f db/functions.sql
psql "$DATABASE_URL" -f db/policies.sql
psql "$DATABASE_URL" -f db/triggers.sql
```

> بعد الاستعادة، أنشئ مستخدمي `auth` من لوحة Supabase (أو استعِد سكيمة `auth`
> من نسخة احتياطية). صفوف `profiles` تُنشأ تلقائياً عبر `handle_new_user`.

## ⚠️ فجوات معروفة (تحتاج توثيقاً تكميلياً — لم تُخترَع هنا)

اكتُشفت أثناء البناء من النواتج المستخرَجة، وهي **خارج نطاق الاستعلامات الستة**
التي بُنِيت منها الملفات. مذكورة هنا صراحةً كي لا تُنسى:

1. **حالة تفعيل RLS لكل جدول** لم تَرِد في النواتج. أضفنا
   `ENABLE ROW LEVEL SECURITY` لكل الجداول الأحد عشر في `schema.sql` لأن وجود
   سياسات RLS يستلزم تفعيلها (وإلا لا تعمل). للتحقق من الحالة الفعلية:
   ```sql
   SELECT relname, relrowsecurity, relforcerowsecurity
     FROM pg_class WHERE relnamespace = 'public'::regnamespace AND relkind = 'r';
   ```

2. ~~مشغّل `auth.users` الذي يشغّل `handle_new_user`.~~ **✅ أُغلقت**: استُخرج
   التعريف الفعلي (`on_auth_user_created`) وأُضيف حرفياً إلى `triggers.sql`.
   يتطلب في الاستعادة الكاملة وجود سكيمة `auth` وجدول `auth.users` (تُنشئهما
   Supabase) قبل تنفيذ `triggers.sql`.

3. **الصلاحيات (GRANTs) على الدوال والجداول** (خصوصاً صلاحية `EXECUTE` لدور
   `authenticated`/`anon` على دوال RPC) لم تَرِد في النواتج — راجعها ووثّقها إن
   لزم لاستعادة كاملة.

4. **إعدادات المصادقة (Auth)** مثل تعطيل التسجيل العام (public signups) وقوالب
   البريد — إعدادات لوحة تحكّم لا تُلتقَط في SQL؛ وثّقها ضمن قائمة تشغيل منفصلة.

> عند الحصول على النواتج التكميلية لهذه الفجوات، أضِفها إلى الملفات المعنية
> (أو ملف جديد مثل `grants.sql`) بنفس مبدأ «حرفياً — بلا اختراع».
